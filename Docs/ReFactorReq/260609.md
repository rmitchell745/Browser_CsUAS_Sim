In `SensorSystem.process`, the engine currently uses Radar dB math for all sensors. 
- Fix: Branch the logic. If `sensor.type === "EO_IR" || sensor.type === "FPV"`, DO NOT use `transmitPowerDb`, `radarSignatureDb`, or `noiseFloorDb`. Instead, optical detection relies strictly on Line-of-Sight and range. 
- Optical Math: `signalStrengthDb` and `noiseDb` are irrelevant. Calculate `confidence = 1 - Math.pow((rangeM / sensor.maxRangeM), 2)`. If `confidence > 0.1`, generate the detection.

Feature Request: Agent-Based Retasking, Sensor Cueing, and Red Mission Heuristics

We need to upgrade the simulation from static waypoints to dynamic, agent-based behaviors. Blue C2 must actively manage its sensors to build confidence, and Red drones must actively pathfind based on mission goals.

Please implement the following logic in the simulation kernel. Provide exact Search/Replace blocks.

1. **Blue C2 Sensor Cueing (The ISR Loop):**
In `C2System.process`, before checking weapons, the C2 Node must evaluate tracks for cognitive gaps.
- **Identify Gaps:** Find any track where `classificationStatus === "Unknown Air Object"` or `identificationStatus === "Unknown"`.
- **Task Sensors:** Find an `Idle` Blue sensor where `canClassify === true` (or `canIdentify === true`). 
- **The Slew:** Update that sensor's `headingDeg` to point at the track's current `posX/posY`. Change the sensor's state to `Slewing` for `sensor.slewRateSec` (if applicable) or `classificationLatency_sec`. 
- **The Result:** The sensor focuses on the track to rapidly increase `trackQuality` and `confidence` until the track is fully Identified, at which point the C2 Node releases the sensor back to `Idle` (spinning/scanning).

2. **Red Mission Heuristics (Autonomous Target Selection):**
Update the `Red Platform Template` schema to include a new object: `missionProfile`.
- `missionProfile.type`: Enum (`Geographic`, `SpecificAsset`, `MaxDamage`).
- `missionProfile.targetTemplateId`: String (Used if `SpecificAsset`).
- In `MovementSystem.process`, if a Red UAS reaches its final static waypoint, it checks its `missionProfile`.
   - If `Geographic`: The drone transitions to `Loiter` at the waypoint until fuel/battery dies.
   - If `SpecificAsset`: The drone scans the Blue Asset list for the specified `targetTemplateId`. It sets its heading toward the closest matching asset and transitions to `Attack Run`.
   - If `MaxDamage`: The drone scans the Blue Asset list, finds the asset with the highest `assetValuePts`, sets its heading toward it, and transitions to `Attack Run`.

3. **Target-Lock Override in Movement (Dogfighting/Pursuit):**
In `MovementSystem.process`, before moving, check if the object has an actively engaging effector (Blue Interceptor) OR is in an `Attack Run` (Red Kamikaze).
- Logic: Find the object's `lockedTargetId` (or the target identified by the Red Heuristic).
- DO NOT use static `waypoints`. Calculate the movement step directly toward the locked target's current `runtime.position` (X, Y, Z). 
- If the locked target is destroyed or the lock drops, the drone resumes its static `waypoints` or `Loiter` behavior.

4. **Log the Behaviors:**
Trigger `services.logger.record` when:
- `[C2 Node] cued [Sensor Name] to investigate [Track ID]`.
- `[Red Drone] initiated autonomous Attack Run against [Blue Asset] based on [MaxDamage] heuristic`.
